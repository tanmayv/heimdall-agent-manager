The experiment

Can tasks and memory reconstruct a product timeline?

Yes. The project history is not just commits. It includes intent, sequencing, blockers, reviewer outcomes, runtime restarts, and the reasoning that shaped the next task.

Executive summary

The project evolved in four arcs

1. POC daemon foundations

Odin began by separating transport, command handling, router, message bus, and durable task behavior.

2. User-facing agent OS

Electron chat, task UI, notifications, unread state, markdown, and runtime restart flows made the system operable.

3. Memory and project intelligence

Approved memory, templates, project anchors, bootstrap context, and memory UI made agents less amnesiac.

4. Rebrand into Heimdall

The product identity shifted from an Odin testbed to Heimdall AI Manager: a connector and manager for agent companies.

Roadblock #1

Runtime visibility was not just code — it was startup ordering

Agents could be connected at the daemon while the UI still failed to display them. The task history captured that the fix was operational: relaunch UI after agents were connected.

Evidence

task-fd7890693916 restarted daemon, agents, and UI. task-a2e6fa9fad5e found UI had connected before agents were successfully started.

Lesson

Task history preserves not only final code but runtime playbooks: check /health, check /clients, clear stale tmux windows, then relaunch UI if needed.

Roadblock #2

Memory correctness required harsh review gates

Atomicity

The reviewer caught that failed durable appends could mutate projection. The fix made durable append happen before projection.

Auth

Direct /memory/* routes were hardened to require valid agent or client credentials.

Scope inheritance

Edit and rollback proposals had to preserve subject/scope so bootstrap could resolve the right active memory.

This is exactly what task+review history is good at: it records not just success, but the failure modes that made the final design safer.

Project intelligence

From cwd to logical project

The project model deliberately avoided overfitting to a single directory. A project is a logical workspace with anchors.

task-dfadea190b97

Implemented durable Project records with project_id, name, description, and loose anchors.

ProjectAnchor

Exactly type, value, and note. Type is arbitrary: directory, git_repo, file, URL, custom.

This matters because future memory and bootstrap resolution can be project-aware without pretending every project is one cwd.

Template memory

A new reviewer should not be born blank

task-f6f8ede4b0ee added template memory so agents can be configured with reusable starter memory.

Approved only

Bootstrap includes approved active templates, not pending/rejected/archived ones.

Agent config

Wrapper/global and per-agent command memory templates can be configured.

Role seeding

Future agents can inherit “reviewer,” “coder,” or “coordinator” defaults without manual retraining.

Chat reliability

Delivery semantics became explicit

task-ee4260b5b6b4 hardened send-to-user: ok=true must mean a persisted, non-empty message id exists.

Synchronous failure

Invalid or unknown user IDs fail before append; no fake success with empty message_id.

Async failure

If WS delivery later fails, daemon records Delivery_Failed with timestamp and error and emits metadata-only notification.

Self-hosting behavior

What is worth highlighting publicly

Four demo-worthy moments

1. Need-improvements loops

Reviewer catches issues, task returns to ready, coder fixes, review validates. This is agent QA, not blind autonomy.

2. Memory bootstrap

New agents can inherit approved memory/templates rather than raw chat history or tribal knowledge.

3. Runtime recovery

Task history reconstructs exact restart and visibility fixes across daemon, wrappers, tmux, agents, UI.

4. Rebrand safety

The rename chain includes compatibility aliases, Home Manager namespace migration, docs, local bin path smokes, and final validation.

Closing thesis

Heimdall turns multi-agent chaos into an accountable system

The project timeline proves the product thesis: when every agent action is tied to tasks, memory, review, and validation, you can reconstruct how a complex system came to exist — and debug where it went wrong.

That is the promise of Heimdall AI Manager: not more agents, but better-managed agents.
